# Design Operating Cadence

This skill produces an operating cadence map — the recurring meetings, rituals, and forums that turn the operating system into rhythm. Each item has a clear purpose; the map is short enough to fit on one screen. Use it when starting a new team, after a reorg, or whenever the current rhythm feels accidental rather than designed.

Part of `org-foundations-and-planning` (Chapter 2 of *Scaling People*).

## Mode

Ask the user which mode to run in. Default to `draft` if they don't specify.

- **`teach`** — Socratic walk-through (10-20 min). Use when the user is new to this artifact or wants to internalize the framework. Walk through every heuristic in the Rubric as a question; build the artifact incrementally; explain the "why" alongside the "what."
- **`draft`** — Targeted intake → first draft → refinement (5-10 min). Use when the user knows roughly what they want. Ask the 3-5 questions listed under "Draft-mode intake" below; produce a complete first draft; offer one round of refinement against the Rubric.
- **`polish`** — Critique-and-polish (2-5 min). Use when the user pastes an existing draft. Critique against every Rubric heuristic; rewrite to address gaps; explain what changed.

## Rubric

1. **Agree on the cadence up front.** Cadences vary by team and should be agreed upon by all relevant parties before they're set — not imposed after the fact. Surprise rhythms breed resentment; co-designed rhythms stick.

2. **Mirror the company cadence, don't fight it.** Your team's cadence will almost certainly be shaped by milestones in the company calendar — annual planning, quarterly business reviews, fiscal year close. Align with these rather than inventing parallel rhythms that create calendar collisions.

3. **Have a cadence and be ready to iterate it.** The key is to have a cadence but not get stuck in it. Plan to revisit your cadence regularly — if a forum is no longer earning its slot, cut it or reshape it before it goes stale.

4. **Each item needs a clear purpose and a single owner.** Ideally there is one owner for each key structure in the cadence, and all other leaders agree to honor the structure. A forum without an owner is a forum on its way to being ignored.

5. **Good process subtracts meetings; it does not add them.** Ask whether any proposed forum's objective could be achieved through a written update or a shared tracker. A good cadence should reduce the total number of meetings needed, not grow it. Process is not synonymous with meetings.

6. **Defend against defensive process.** When ownership of decisions is unclear, teams create cover-your-ass processes — reviews that multiply, approvers that accumulate. Design your cadence around clear decision owners, not around the anxiety of unclear ownership.

7. **Experiment with explicit parameters.** If you're unsure whether a forum will work, run it as an experiment: state how long you'll try it, when you'll check in, and what success looks like. Experimenting with process is professional; what's unprofessional is changing things constantly without evaluation criteria.

## Source

*Scaling People* — Claire Hughes Johnson, Stripe Press 2023 — Chapter 2, "Operating cadence" and "A brief note on process."

Encoded heuristic: An operating cadence is the schedule or rhythm with which a team provides progress reports and makes decisions. Because humans are creatures of habit, the cadence should be stable and predictable — but not so rigid that it can't be iterated. Each forum should have a clear purpose and a single owner. The deeper risk in cadence design is defensive process: when decision ownership is unclear, approvals multiply and meetings accumulate to cover for the ambiguity. A well-designed cadence has the opposite effect — it should reduce the number of ad hoc meetings needed, not increase them. Good process creates lightweight checks that solidify alignment; it is not synonymous with meetings.
